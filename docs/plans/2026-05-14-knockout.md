# FIFA Tournament — Knockout Vertical Slice

**Goal:** Single-elimination knockout end-to-end. Reproducible random seeding via stored `rng_seed`. Byes pre-placed (no fake walkover matches). Winner-propagation on score submission. Tournament auto-completes when final is reported.

## Math recap

For N participants:
- B = next power of 2 ≥ N
- byes = B − N
- R = log2(B) rounds total
- Round-1 matches = (N − byes) / 2 = (2N − B) / 2
- Total matches across all rounds = N − 1 (every match eliminates one participant; champion is the only survivor)

Examples:
| N  | B  | byes | R1 matches | Total |
|----|----|------|------------|-------|
| 2  | 2  | 0    | 1          | 1     |
| 3  | 4  | 1    | 1          | 2 (R1=1, F=1; top seed gets bye to final) |
| 4  | 4  | 0    | 2          | 3     |
| 5  | 8  | 3    | 1          | 4     |
| 7  | 8  | 1    | 3          | 6     |
| 8  | 8  | 0    | 4          | 7     |
| 9  | 16 | 7    | 1          | 8     |
| 16 | 16 | 0    | 8          | 15    |
| 17 | 32 | 15   | 1          | 16    |

### 2. Pure bracket engine (TDD): `internal/knockout/bracket.go` + `bracket_test.go`
- `BuildBracket(participants []uuid.UUID, seed int64) Bracket` — returns ordered list of `BracketMatch{Round, Ord, Home, Away (nullable), NextMatchOrdInRound, NextMatchSlot, Status}`.
- Algorithm:
  1. Shuffle participants with deterministic RNG (rand.New(rand.NewSource(seed))).
  2. Assign seeds 1..N in shuffled order; bracket size B = next pow2.
  3. Place seeds into bracket positions using standard bracket order so seed i meets B+1-i in R1.
  4. Top `byes` seeds (1..byes) skip R1 and go directly into the corresponding R2 slot.
  5. Generate R1 matches for the remaining 2*(N-byes)/2 participant pairs.
  6. Generate R2..RR placeholder matches and wire `next_match` links so each match knows where its winner lands.
- `PropagateWinner(matches []Match, completedID uuid.UUID, winnerID uuid.UUID) (updates []SlotUpdate, finalCompleted bool, err error)` — pure helper that returns intended writes.
- Tests for N∈{2,3,4,5,7,8,9,16,17}: assert match count = N−1, every leaf reachable to final, deterministic with fixed seed.

### 3. Store wiring
- `StartKnockout(ctx, tournamentID, seed, bracketMatches)` — inside tx: set tournament ACTIVE + version+1, insert matches with proper status, then UPDATE second pass to fill `next_match_id` (since the placeholder rows need IDs). Insert deterministic ord so we can resolve next_match_id by `(round, ord)` lookup.
- Extend `Match` struct with `NextMatchID *uuid.UUID`, `NextMatchSlot *string`, nullable participant IDs.
- `SubmitScoreKnockout(ctx, matchID, home, away, expectedVersion)`:
  - Reject draws.
  - tx: lock target match (`FOR UPDATE`), verify version, write score + status=COMPLETED, version+1.
  - Compute winner participant id.
  - If `next_match_id` is set: lock that downstream match. If it is `COMPLETED` → return `ErrDownstreamCompleted`. Else write winner into correct slot; if both slots filled, status=`PLAYABLE`; version+1.
  - If this is the final (no next match): mark tournament COMPLETED.
  - Return updated match.

### 6. Golden path script: `scripts/golden-path-knockout.sh`
- Register 5 users, each picks a unique team.
- User 1 creates KO tournament.
- All 5 join.
- Start. Verify detail: 4 total matches, R1 has 1, R2 has 2, R3 has 1, exactly 3 byes pre-placed into R2 slots.
- Score R1. Score the two R2 matches. Score final. Verify tournament COMPLETED and champion correct.
- Negative: try to edit R1 after its R2 child is COMPLETED → expect 409. Try to submit a draw → expect 422.

## Open questions / scope cuts

- No re-seeding after a participant drops mid-tournament (out of scope).
- No third-place playoff.
- "Reset match" UX is out of scope — the error message tells the user the system needs an admin reset; we don't expose a reset endpoint this iteration.
- Draws in knockout are simply rejected; we don't implement penalty shootout UI.
